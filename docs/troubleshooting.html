📖 مقدمة

يهدف هذا الدليل إلى مساعدة المسؤولين والمستخدمين في تشخيص وحل المشكلات الشائعة التي قد تواجههم أثناء تثبيت أو تشغيل أو استخدام نظام المحاسبة المتكامل. يغطي الدليل الأخطاء الأكثر تكراراً مع الحلول المقترحة وأوامر الإصلاح اللازمة.

  • شرح تفصيلي لأسباب كل خطأ وأعراضه
  • حلول سريعة مع أوامر جاهزة للنسخ واللصق
  • جداول مرجعية سريعة لتشخيص الأخطاء الشائعة
  • نصائح لتجنب المشاكل قبل حدوثها
ⓘ
ملاحظة مهمة قبل تطبيق أي حل، تأكد من عمل نسخة احتياطية من قاعدة البيانات والملفات الإعدادية. بعض الحلول قد تتطلب تدخلاً يدوياً في قاعدة البيانات.

⚙ 1. مشاكل التثبيت والإعداد

تظهر هذه المشاكل عادةً أثناء التثبيت الأولي للنظام أو عند تغيير بيئة التشغيل. تأكد من اتباع خطوات التثبيت بدقة قبل البحث عن حلول أخرى.

1.1 خطأ DJANGO_SECRET_KEY is required

ImproperlyConfigured: DJANGO_SECRET_KEY is required
السبب: لم يتم تعريف متغير البيئة DJANGO_SECRET_KEY في ملف البيئة أو في نظام التشغيل. هذا المفتاح ضروري لعمليات التشفير وحماية الجلسات.
✓ الحل:
  1. افتح ملف .env في جذر المشروع
  2. أضف المفتاح التالي بقيمة قوية عشوائية:
$ python -c "from django.core.management.utils import get_random_secret_key; print(get_random_secret_key())"
# أضف النتيجة في ملف .env DJANGO_SECRET_KEY=your-generated-secret-key-here

1.2 خطأ اتصال قاعدة البيانات

رسالة الخطأ السبب الحل الخطورة
connection refused PostgreSQL غير قيد التشغيل أو المنفذ خاطئ تشغيل خدمة PostgreSQL والتحقق من المنفذ (5432 افتراضياً) حرج
database does not exist قاعدة البيانات لم يتم إنشاؤها بعد إنشاء قاعدة البيانات يدوياً أو تشغيل المهاجرات عالي
authentication failed اسم المستخدم أو كلمة المرور غير صحيحة التحقق من إعدادات DATABASES في settings.py عالي
password authentication failed كلمة مرور المستخدم خاطئة في pg_hba.conf تحديث إعدادات المصادقة في PostgreSQL متوسط
✓ خطوات الحل:
# 1. التحقق من تشغيل PostgreSQL $ pg_isready -h localhost -p 5432 # 2. إنشاء قاعدة البيانات $ createdb accounting_system # 3. تشغيل المهاجرات $ python manage.py migrate

1.3 خطأ ModuleNotFoundError

الوحدة المفقودة السبب الحل الخطورة
django / rest_framework / وحدات أخرى المكتبة غير مثبتة في البيئة الافتراضية pip install -r requirements.txt عالي
psycopg2 مكتبة libpq-dev غير مثبتة (Linux) sudo apt install libpq-dev python3-dev عالي
psycopg2-binary بديل لـ psycopg2 بدون اعتمادات نظام pip install psycopg2-binary متوسط
django_filters مكتبة تصفية غير مثبتة pip install django-filter متوسط
celery / redis مكتبات معالجة المهام الخلفية pip install celery redis منخفض
⚠
نصيحة يُفضل دائماً تثبيت المكتبات داخل بيئة افتراضية (virtual environment) لتجنب تضارب الإصدارات مع حزم النظام.

⚡ 2. مشاكل التشغيل

تشمل المشاكل التي تظهر أثناء تشغيل النظام وydao السيرفر، مثل عدم استجابة السيرفر أو ظهور صفحات خطأ.

2.1 السيرفر لا يستجيب

الأعراض: المتصفح يعرض "This site can't be reached" أو ينتظر طويلاً بدون استجابة.
  1. التحقق من أن السيرفر قيد التشغيل:
# فحص المنفذ 8012 $ netstat -ano | findstr :8012 # أو على Linux/Mac $ ss -tlnp | grep 8012
  1. فحص ملفات السجلات:
# مراجعة السجلات $ tail -f logs/accounting.log # على Windows $ Get-Content logs/accounting.log -Wait
  1. إعادة تشغيل السيرفر:
# إيقاف العمليات القديمة ثم إعادة التشغيل $ taskkill /F /IM python.exe 2>nul $ python manage.py runserver 0.0.0.0:8012

2.2 صفحة 500 - خطأ داخلي في السيرفر

السبب المحتمل السبب الحل الخطورة
قاعدة البيانات غير مهاجرة الجداول الجديدة لم يتم إنشاؤها python manage.py migrate حرج
ملف ثابت مفقود ملفات CSS/JS غير موجودة python manage.py collectstatic عالي
خطأ في الكود استثناء غير معالج في الكود فحص السجلات وتصحيح الكود حرج
DEBUG = False بدون ALLOWED_HOSTS السيرفر في وضع الإنتاج بدون إعدادات إضافة اسم المضيف في ALLOWED_HOSTS عالي
✓ خطوات التشخيص:
# 1. فحص شامل للإعدادات $ python manage.py check # 2. عرض أخطاء المهاجرة $ python manage.py showmigrations # 3. تطبيق المهاجرات المعلقة $ python manage.py migrate # 4. جمع الملفات الثابتة $ python manage.py collectstatic --noinput

2.3 صفحة 404 - الصفحة غير موجودة

الأعراض: يعرض المتصفح "Page not found" عند محاولة الوصول لصفحة محددة.
✓ الحلول المحتملة:
  1. التحقق من صحة المسار في عنوان URL والتأكد من عدم وجود أخطاء إملائية
  2. التأكد من أن الوحدة (app) مثبتة في INSTALLED_APPS في ملف settings.py
  3. التحقق من تعريف المسارات في ملف urls.py الرئيسي وملفات urls.py الخاصة بكل وحدة
  4. تشغيل الأمر التالي لعرض جميع المسارات المسجلة:
# عرض جميع المسارات المسجلة $ python manage.py show_urls

🗃 3. مشاكل قاعدة البيانات

تشمل مشاكل المهاجرات والجداول والأعمدة و CoreData المفقودة. هذه المشاكل شائعة خاصة أثناء التحديثات.

3.1 جدول مفقود (relation does not exist)

django.db.utils.OperationalError: relation "appname_tablename" does not exist
السبب: جدول لم يتم إنشاؤه بسبب فشل في المهاجرة أو تخطي خطوة من خطوات المهاجرة.
✓ الحلول:
# الحل 1: تطبيق جميع المهاجرات المعلقة $ python manage.py migrate # الحل 2: إذا كان الملف المهاجر موجود لكن الجدول لم يتم إنشاؤه $ python manage.py migrate --fake # الحل 3: إعادة تطبيق مهاجرات وحدة محددة $ python manage.py migrate app_name

3.2 خطأ no such column

OperationalError: no such column: appname.fieldname
السبب: لم يتم تطبيق الـ migration الذي يضيف العمود الجديد، أو تم تثبيت قاعدة بيانات قديمة.
✓ الحل:
# تطبيق مهاجرات وحدة محددة $ python manage.py migrate app_name # أو تطبيق جميع المهاجرات $ python manage.py migrate # عرض حالة المهاجرات للتشخيص $ python manage.py showmigrations
❗
تحذير خطير لا تستخدم --fake إلا إذا كنت متأكداً من أن الجدول أو العمود موجود فعلاً في قاعدة البيانات. استخدامه بشكل خاطئ قد يؤدي لفقدان بيانات.

3.3 قاعدة البيانات مقفلة (database is locked)

django.db.utils.OperationalError: database is locked
السبب: SQLite لا يدعم الوصول المتزامن بشكل جيد، أو هناك عملية طويلة تشغل القاعدة.
✓ الحلول:
  1. في SQLite، زيادة مهلة الانتظار:
# في settings.py، أضف OPTIONS DATABASES = { 'default': { 'ENGINE': 'django.db.backends.sqlite3', 'NAME': BASE_DIR / 'db.sqlite3', 'OPTIONS': { 'timeout': 30, } } } # أو عبر SQL مباشر: PRAGMA busy_timeout=30000;
  1. التأكد من عدم تشغيل عدة نسخ من السيرفر في نفس الوقت
  2. انتظار اكتمال العمليات الطويلة ثم إعادة المحاولة
  3. الانتقال إلى PostgreSQL للإنتاج (يوصى به بشدة)

جدول مرجعي سريع - أخطاء قاعدة البيانات

رسالة الخطأ النوع الأولوية الحل السريع
relation "..." does not exist جدول مفقود حرج python manage.py migrate
no such column: ... عمود مفقود عالي python manage.py migrate app_name
database is locked قفل الوصول متوسط زيادة timeout / التحقق من العمليات النشطة
column "..." of relation "..." already exists عمود مكرر عالي python manage.py migrate app_name 000X --fake
duplicate key value violates unique constraint قيمة مكررة متوسط تنظيف البيانات المكررة يدوياً

👤 4. مشاكل المستخدمين والصلاحيات

4.1 تسجيل الدخول لا يعمل

المشكلة السبب الحل
نسيت كلمة المرور المستخدم لا يتذكر كلمة المرور python manage.py changepassword username
الحساب غير نشط isActive = False تفعيل الحساب من لوحة الإدارة أو الأمر
حساب غير موجود المستخدم لم يتم إنشاؤه python manage.py createsuperuser
خطأ في المصادقة إعدادات AUTH غير صحيحة التحقق من AUTH_USER_MODEL في settings.py
✓ أوامر مساعدة:
# إعادة تعيين كلمة المرور $ python manage.py changepassword admin # تفعيل حساب مستخدم $ python manage.py shell >>> from django.contrib.auth.models import User >>> u = User.objects.get(username='admin') >>> u.is_active = True >>> u.save() # إنشاء مستخدم جديد (Superuser) $ python manage.py createsuperuser

4.2 صلاحية غير كافية (Permission Denied)

الأعراض: يعرض النظام صفحة "403 Forbidden" أو رسالة "You don't have permission to perform this action".
✓ الحلول:
  1. تواصل مع مسؤول النظام لمنح الصلاحية المطلوبة
  2. أضف الصلاحية من لوحة الإدارة Django Admin في مسار /admin/
  3. تأكد من أن المجموعة (Group) للمستخدم تحتوي على الصلاحيات الصحيحة
  4. للإضافة السريعة لصلاحية عبر الأمر:
# منح صلاحية لمستخدم $ python manage.py shell >>> from django.contrib.auth.models import User, Permission >>> user = User.objects.get(username='username') >>> perm = Permission.objects.get(codename='add_account') >>> user.user_permissions.add(perm) >>> user.save()

📋 5. مشاكل Docker

5.1 خطأ port already in use

Bind for 0.0.0.0:8012 failed: port is already allocated
السبب: المنفذ المطلوب مستخدم من قبل خدمة أخرى أو حاوية Docker قديمة.
✓ الحلول:
  1. تغيير المنفذ في ملف .env:
# في ملف .env PORT=8013 # غيّر من 8012 إلى 8013
  1. إيقاف الحاوية القديمة:
# إيقاف وحذف الحاويات $ docker compose down # أو حذف الحاوية يدوياً $ docker stop container_name $ docker rm container_name
  1. إيقاف الخدمة المستخدمة على المنفذ (Windows):
$ netstat -ano | findstr :8012 $ taskkill /PID <PID> /F

5.2 خطأ container exited with code 1

السبب: الحاوية توقفت بسبب خطأ في التهيئة، عادةً بسبب متغيرات بيئة غير صحيحة أو ملفات مفقودة.
✓ خطوات التشخيص:
# عرض سجلات الحاوية $ docker compose logs web # عرض السجلات بشكل مستمر $ docker compose logs -f web # عرض الحاويات المتوقفة $ docker ps -a | findstr Exited
ⓘ
الأسباب الأكثر شيوعاً: تحقق أولاً من ملف .envتأكد من صحة جميع المتغيرات尤其是 DJANGO_SECRET_KEY و DATABASE_URL و POSTGRES_PASSWORD.

جدول مرجعي - أخطاء Docker الشائعة

رسالة الخطأ السبب الحل الخطورة
port is already allocated المنفذ مستخدم تغيير المنفذ أو إيقاف الخدمة القديمة متوسط
container exited with code 1 خطأ في التهيئة فحص logs و متغيرات البيئة عالي
no space left on device مساحة القرص ممتلئة docker system prune -a عالي
cannot connect to the Docker daemon Docker Desktop غير قيد التشغيل تشغيل Docker Desktop حرج
network ... not found شبكة Docker غير موجودة docker compose up --build متوسط

🚀 6. مشاكل الأداء

6.1 بطء في تحميل الصفحات

✓ حلول تحسين الأداء:
  1. تفعيل التخزين المؤقت (Redis Cache):
# في settings.py CACHES = { 'default': { 'BACKEND': 'django.core.cache.backends.redis.RedisCache', 'LOCATION': 'redis://127.0.0.1:6379/1', } }
  1. زيادة عدد عمليات Gunicorn:
# زيادة عدد workers (القاعدة: 2 × عدد الأنوية + 1) $ gunicorn --workers 4 --bind 0.0.0.0:8012 config.wsgi:application
  1. إضافة فهارس (indexes) لقاعدة البيانات:
# في models.py، أضف db_index=True للأعمدة المُستخدمة في البحث class Account(models.Model): code = models.CharField(max_length=50, db_index=True) name = models.CharField(max_length=200, db_index=True)
  1. استخدام select_related و prefetch_related لتحسين استعلامات ORM:
# بدلاً من: entries = JournalEntry.objects.all() for entry in entries: print(entry.account.name) # N+1 query # استخدم: entries = JournalEntry.objects.select_related('account').all()

6.2 استهلاك عالي للذاكرة

✓ الحلول:
  1. تقليل عدد Gunicorn workers إذا كانت الذاكرة محدودة
  2. تفعيل Pagination في جميع القوائم:
# في settings.py DEFAULT_PAGINATION = 25 # في views.py from django.core.paginator import Paginator paginator = Paginator(queryset, 25)
  1. استخدام iterator() للبيانات الكبيرة:
# بدلاً من تحويل كل البيانات للذاكرة for entry in JournalEntry.objects.iterator(chunk_size=1000): process(entry)

💾 7. مشاكل الباك أب

7.1 فشل عملية الباك أب

المشكلة السبب الحل الخطورة
No space left on device مساحة القرص غير كافية التحقق من المساحة المتاحة وحذف الباك أبات القديمة حرج
Permission denied عدم صلاحيات الكتابة تعديل صلاحيات مجلد الباك أب عالي
Database connection failed انقطاع الاتصال أثناء الباك أب التأكد من استقرار اتصال قاعدة البيانات عالي
Backup file corrupted توقف العملية في منتصفها إعادة تشغيل الباك أب والتحقق من سلامة الملف حرج
✓ أوامر الباك أب الصحيحة:
# PostgreSQL $ pg_dump -U username -h localhost accounting_db > backup_$(date +%Y%m%d).sql # SQLite $ cp db.sqlite3 backup_$(date +%Y%m%d).sqlite3 # استعادة PostgreSQL $ psql -U username accounting_db < backup_file.sql # استعادة SQLite $ cp backup_file.sqlite3 db.sqlite3
⚠
توصية أمنية لا ت تخزّن ملفات الباك أب على نفس السيرفر. انسخها إلى موقع خارجي آمن بشكل دوري.

📞 8. أرقام الدعم الفني وحالات التصعيد

إذا لم تتمكن من حل المشكلة باستخدام الحلول المذكورة أعلاه، اتبع خطوات التصعيد التالية:

خطوات التشخيص الأولية

# 1. جمع معلومات النظام $ python manage.py check --deploy # 2. عرض إصدارات المكتبات $ pip freeze > requirements_snapshot.txt # 3. فحص حالة السيرفر $ python manage.py showmigrations # 4. تصدير السجلات $ cp logs/accounting.log logs/error_$(date +%Y%m%d_%H%M%S).log
