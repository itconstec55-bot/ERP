الباب الأول: نقل البيانات والترحيل

1.1 تحديد المتطلبات قبل النقل

أجهزة المصدر والهدف

البند الجهاز المصدر (القديم) الجهاز الهدف (الجديد)
نظام التشغيل Windows 10/11 Windows 10/11 أو Server 2022
المعالج 2 نواة على الأقل 4 نواة أو أكثر
الرام 4 GB كحد أدنى 8 GB أو أكثر
التخزين 50 GB متاح 100 GB متاح (SSD مفضل)
الشبكة اتصال LAN أو USB اتصال LAN أو USB

أدوات النقل المطلوبة

الأداة الغرض كيفية الحصول عليها
WinRAR أو 7-Zip فك الضغط 7-zip.org
USB Flash 16GB+ النقل اللامكاني أي محل إلكترونيات
أو Share على الشبكة النقل عبر LAN مدمج في Windows
PowerShell 5.1+ سكريبتات الإعداد مدمج في Windows

فحص الجهاز القديم قبل النقل

1. افتح CMD كمسؤول ونفذ:
systeminfo | findstr /B /C:"OS Name" /C:"Total Physical Memory" /C:"Available Physical Memory"

2. تأكد من مساحة القرص المتاحة:
wmic logicaldisk get size,freespace,caption

3. تأكد من إصدار Python:
python --version

4. تأكد من حجم قاعدة البيانات:
dir D:\accounting_system\db.sqlite3

1.2 إنشاء النسخة الاحتياطية الكاملة

الملفات التي يجب تضمينها

المكون المسار الحجم التقريبي
كود التطبيق D:\accounting_system\ (كل المجلدات عدا venv) 800 MB
قاعدة البيانات D:\accounting_system\db.sqlite3 200 MB
ملفات الإعدادات .env, requirements.txt 1 MB
المجلد الثابت static/, staticfiles/ 500 MB
المجلد المرفوع media/ 500 MB
القوالب templates/ 20 MB
التوثيق docs/, README.md 5 MB

إجراء الإنشاء

على الجهاز القديم:

# فتح PowerShell كمسؤول والانتقال للمجلد
cd D:\accounting_system

# تشغيل سكريبت النسخ الاحتياطي
powershell -ExecutionPolicy Bypass -File create_full_backup.ps1

الناتج المتوقع:

D:\accounting_system_backups\accounting_system_FULL_[التاريخ].zip
الحجم: حوالي 1 GB
عدد الملفات: حوالي 966 ملف

الملفات المستبعدة من النسخة الاحتياطية

الملف/المجلد السبب
venv/ البيئة الافتراضية (يتم إنشاؤها من جديد)
__pycache__/ ملفات بايثون المؤقتة
.git/ سجل Git (يتم نقله بشكل منفصل إذا لزم)
*.pyc ملفات الكود المُجمّع
logs/ سجلات التشغيل القديمة
backups/ نسخ احتياطية سابقة
*.sqlite3-shm ملفات SQLite المؤقتة
*.sqlite3-wal ملفات SQLite اليومية

1.3 فحص سلامة النسخة الاحتياطية

1. تحقق من وجود الملف:
   dir D:\accounting_system_backups\accounting_system_FULL_*.zip

2. تحقق من الحجم (يجب أن يكون أكبر من 800 MB):
   forfiles /P D:\accounting_system_backups /M /C "cmd /c if @fsize GTR 838860800 echo @fname OK"

3. اختبار فك الضغط بدون تثبيت:
   using 7-Zip:
   "C:\Program Files\7-Zip\7z.exe" t D:\accounting_system_backups\accounting_system_FULL_*.zip

1.4 نقل الملفات إلى الجهاز الهدف

الخيار أ: عبر الشبكة (LAN)

على الجهاز المصدر:

1. افتح الاستخدام المشترك:
  右键 على المجلد D:\accounting_system → Properties → Sharing → Advanced Sharing
   → Share this folder → Permissions → Everyone → Full Control

2. احصل على IP الجهاز المصدر:
   ipconfig
   مثال: 192.168.1.50

3. على الجهاز الهدف، افتح:
   \\192.168.1.50\accounting_system_backups
   وانسخ ملف ZIP إلى سطح المكتب

الخيار ب: عبر USB

1. انسخ ملف ZIP إلى USB:
   copy D:\accounting_system_backups\accounting_system_FULL_*.zip E:\

2. اخرج الـ USB بأمان:
   Remove Hardware Safely → USB Flash Drive

3. على الجهاز الهدف، الصق الـ USB وانسخ الملف إلى D:\

الخيار ج: عبر الإنترنت (mega, google drive, etc.)

1. ارفع الملف إلى الخدمة السحابية
2. على الجهاز الهدف، حمّله
3. تأكد من صلاحية الحمولة باستخدام: sha256sum

1.5 فك الضغط وتوسيع المجلدات

على الجهاز الهدف:

# 1. إنشاء المجلد الهدف
mkdir D:\accounting_system

# 2. فك الضغط
# باستخدام PowerShell المدمج:
Expand-Archive -Path "D:\accounting_system_FULL_*.zip" -DestinationPath "D:\accounting_system" -Force

# أو باستخدام 7-Zip:
"C:\Program Files\7-Zip\7z.exe" x D:\accounting_system_FULL_*.zip -oD:\accounting_system -y

1.6 فحص سلامة الملفات بعد النقل

# 1. تحقق من وجود الملفات الرئيسية
@(

(Get-ChildItem D:\accounting_system -Recurse -File).Count

3.5 النقاط الحرجة عند تغيير البيئة

النقطة التوضيح الحل
كلمة المرور تختلف بين البيئات استخدم .env منفصل لكل بيئة
قاعدة البيانات SQLite لا تعمل مع كثرة المستخدمين استخدم PostgreSQL للإنتاج
HTTPS مطلوب في الإنتاج استخدم Let's Encrypt أو شهادة مدفوعة
static files يجب تجميعها يدوياً python manage.py collectstatic
DEBUG يكشف الأخطاء للمستخدمين اجعله False في الإنتاج
ALLOWED_HOSTS يمنع الهجمات حدد النطاقات فقط
النسخ الاحتياطي يجب أن يكون تلقائياً استخدم Windows Task Scheduler

4.1 نموذج المزامنة المركزي (Central Database)

المبدأ

جميع الأجهزة تتصل بقاعدة بيانات واحدة على سيرفر مركزي.

الم架构

┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  الجهاز 1    │────▶│              │◀────│  الجهاز 2    │
│  (القديم)    │     │  سيرفر DB    │     │  (الجديد)    │
│              │◀────│  PostgreSQL   │────▶│              │
└──────────────┘     └──────────────┘     └──────────────┘

إعداد قاعدة البيانات المركزية

# على جهاز السيرفر:
cd D:\accounting_system
powershell -ExecutionPolicy Bypass -File setup_postgresql_server.ps1

# الناتج:
# IP: 192.168.1.100
# Port: 5432
# Database: accounting_db
# User: accounting_user
# Password: [محفوظ]

تحديث .env على كل جهاز

# على كل جهاز:
cd D:\accounting_system
powershell -ExecutionPolicy Bypass -File update_env_postgresql.ps1
# غيّر:
#   $DB_SERVER_IP = "192.168.1.100"
#   $DB_PASSWORD = "نفس_الباسوورد"

نقل البيانات مرة واحدة

# على الجهاز القديم فقط:
powershell -ExecutionPolicy Bypass -File migrate_sqlite_to_postgresql.ps1

4.2 نموذج المزامنة المعمارية (Master-Slave)

المبدأ

جهاز واحد هو المصدر (Master) والأجهزة الأخرى نسخ (Slaves).

إعداد PostgreSQL للنسخ

-- على الـ Master:
-- تعديل postgresql.conf
wal_level = replica
max_wal_senders = 3
wal_keep_size = 1024

-- تعديل pg_hba.conf
host replication slave_user 192.168.1.0/24 scram-sha-256

-- إنشاء مستخدم النسخ
CREATE ROLE slave_user WITH REPLICATION LOGIN PASSWORD 'password';

إعداد الـ Slave

# 1. توقف PostgreSQL على الـ Slave
net stop postgresql-x64-16

# 2. حذف البيانات القديمة
Remove-Item -Recurse -Force "C:\Program Files\PostgreSQL\16\data"

# 3. النسخ من الـ Master
pg_basebackup -h 192.168.1.100 -U slave_user -D "C:\Program Files\PostgreSQL\16\data" -Fp -Xs -P -R

# 4. تشغيل الـ Slave
net start postgresql-x64-16

4.3 نموذج المزامنة ثنائية الاتجاه (Bidirectional)

باستخدام Django + PostgreSQL

# في settings.py - إعدادات المزامنة
DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.postgresql',
        'NAME': 'accounting_db',
        'USER': 'accounting_user',
        'PASSWORD': os.environ.get('DB_PASSWORD'),
        'HOST': os.environ.get('DB_HOST', 'localhost'),
        'PORT': '5432',
        'OPTIONS': {
            'connect_timeout': 10,
            'options': '-c statement_timeout=30000'
        }
    }
}

# إعداد اللوجز للتتبع
LOGGING = {
    'version': 1,
    'handlers': {
        'sync_file': {
            'level': 'INFO',
            'class': 'logging.FileHandler',
            'filename': 'logs/sync.log',
        },
    },
    'loggers': {
        'sync': {
            'handlers': ['sync_file'],
            'level': 'INFO',
            'propagate': False,
        },
    },
}

الباب الخامس: الميزات التقنية المتقدمة (الإصدار 2.2)

فيما يلي الميزات الجديدة التي أُضيفت في الإصدار 2.2 مع شرح تأثيرها التقني:

5.1 FIFO Costing — تكلفة المخزون بالوارد أولاً

التأثير التقني: كل حركة مخزون تُسجّل كطبقة مستقلة. عند الخروج، تُخصم من أقدم طبقة أولاً.

  • النمط: طبقات مخزون (Inventory Layers)
  • الملفات: inventory/models/fifo.py, inventory/services/fifo_engine.py
  • الاختبارات: اختبارات مخصصة لـ FIFO (تأكيد تسلسل التكلفة)

5.2 OpenAPI + Swagger + ReDoc

التأثير التقني: توليد تلقائي لتوثيق API باستخدام drf-spectacular.

  • مكتبة: drf-spectacular — تولد مخطط OpenAPI 3.0
  • ثلاث واجهات: /schema/ (JSON/YAML), /docs/ (Swagger UI), /redoc/ (ReDoc)
  • يدعم authentication عبر Session + Token

5.3 HTMX + Alpine.js

التأثير التقني: تحميل جزئي للصفحات وتفاعلات أمامية بدون jQuery.

  • HTMX: يرسل طلبات AJAX تلقائياً مع hx-* attributes في HTML
  • Alpine.js: ربط بيانات خفيف (إظهار/إخفاء، عدادات، نماذج)
  • Form Validation (JS): تحقق آني client-side في static/js/form-validation.js
  • يُعاد تفعيل HTMX بعد التحميل الديناميكي تلقائياً

5.4 PWA / Offline Support

التأثير التقني: تطبيق يعمل دون اتصال عبر Service Worker و Cache API.

  • Service Worker: static/sw.js — استراتيجية Cache First للملفات الثابتة
  • Manifest: static/manifest.json — أيقونة + اسم لتثبيت التطبيق
  • Offline page: صفحة مخصصة عند فقدان الاتصال

5.5 Redis Caching — @cache_page على 9 views

التأثير التقني: تخزين مؤقت للتقارير الثقيلة باستخدام Redis 7 كـ cache backend.

  • الـ views المخبّأة: Dashboard (5 د), تقارير المبيعات (4 تقارير × 10 د), تقارير المشتريات (3 تقارير × 10 د), API endpoints (5 د), التسوية البنكية (10 د)
  • الإجمالي: 9 views مع @cache_page
  • Redis يعمل ضمن Docker Compose مع health check

5.6 N+1 Query Optimization

التأثير التقني: القضاء على مشكلة N+1 في استعلامات قاعدة البيانات.

  • select_related: JOIN مسبق لـ ForeignKey في ViewSets
  • prefetch_related: دفعة منفصلة لـ ManyToMany
  • annotate: تجميع الحسابات في SQL بدلاً من Python
  • Prefetch: تحكم دقيق بالتصفية المسبقة

5.7 أمان متعدد الطبقات — CSP + Idle Timeout + Rate Limiting

التأثير التقني: ثلاث طبقات أمان جديدة مدمجة كـ middleware في Django.

  • CSP Middleware: يمنع XSS — يحدد Content-Security-Policy header مع allowed sources
  • Idle Session Timeout: IDLE_TIMEOUT_MINUTES=30 — يقطع الجلسة بعد 30 دقيقة خمول
  • Rate Limiting: 100 req/min للـ API — يمنع هجمات القوة العمياء
  • الإعدادات في .env: CSP_ENABLED, RATE_LIMIT_ENABLED, IDLE_TIMEOUT_MINUTES

5.8 أدوات DevOps — Coverage + Pre-commit + pytest.ini + pyproject.toml

التأثير التقني: أدوات جودة الكود والاختبارات لضمان الاستقرار.

  • Coverage.py: إعدادات [tool.coverage.run] في pyproject.toml مع fail_under=40 و coverage html للتقرير
  • Pre-commit: .pre-commit-config.yaml — Ruff + formatting + فحص تلقائي قبل كل commit
  • pytest.ini موحّد: testpaths يغطي جميع التطبيقات الـ 22 مع DJANGO_SETTINGS_MODULE
  • pyproject.toml: تكوين مركزي لـ Ruff + pytest + coverage + type hints
  • Factory Boy: مصانع اختبار قابلة لإعادة الاستخدام + 158 اختباراً PASS

5.9 Docker Compose + Health Checks

التأثير التقني: نشر متكامل عبر Docker مع مراقبة صحة الحاويات.

  • 3 حاويات: db (PostgreSQL 16), redis (Redis 7), web (Gunicorn)
  • Health checks لكل حاوية — إعادة تشغيل تلقائي عند الفشل
  • Volumes للبيانات: postgres_data, redis_data
  • 4 أنماط تشغيل: Dev / Production / Service / Docker

5.10 Celery / Eager Mode

التأثير التقني: Task queue للمهام الخلفية مع دعم الوضع المباشر للتطوير.

  • إعدادات Celery في accounting_system/celery.py
  • DJANGO_CELERY_EAGER=true — تنفيذ المهام مباشرة بدون وسيط
  • مناسب للمهام: إرسال تقارير مجدولة، معالجة دفعات، نسخ احتياطي
